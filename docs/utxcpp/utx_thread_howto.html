utx::thread utx::jthread HowTo

Utxcpp utx::thread utx::jthread How-To, c++.

Inheritance

First step, it's to be derived from utx::runnable/utx::jrunnable or utx::thread/utx::jthread,

The first template parameter is the final derivation thread class.

For example: 

class my_thread: virtual public utx::thread<my_thread>;

override pure virtual method

The only method you have to override is the .run() method, which is used to run real threading code. 

{
	...
public:
	void run() override
	{
		// do real things.
	}
	...
};

Although the .run method should not be called by user code, it should be still public, because it will be called by utxcpp thread api method, which is not in the same hierarchical object as the object which declares it.

Calling .run() from user code is not an error, but it makes little sense.

declare object with your class

Now the thread class is functional, everything is OK. You can declare object with it now. 

my_thread thrd;

start the thread

The thread will be not started, if you do not start it by calling .start() from user code. 

thrd.start();

.start() can not be called from utxcpp thread api, it must be called from user code. And it is final, can not be overrided.

.join() method

.join() is to let the parent wait and let the sub-thread to join the main thread.

.join() method will be called automatically by destructor, you do not have to call it again, but you can call it too to let .join() to be called ahead of end.

.join() will first check status by calling .joinable(), then call native thread (std::thread or std::jthread) .join(), so you do not have to check thread status.

Complete cpp code example

Complete c++ code example 

#include <utxcpp/thread.hpp>
#include <utxcpp/core.hpp>

class loop_i_thread: virtual public utx::thread<loop_i_thread>
{
public:
	void run() override
	{
		for (int i=0; i<20; ++i)
			utx::print("i=>", i);
	}
};

int main()
{
	loop_i_thread thread;
	thread.start();
}
